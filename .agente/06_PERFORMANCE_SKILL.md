# 06_PERFORMANCE_SKILL

## Objective

This skill defines the global performance optimization, rendering efficiency, scalability and resource management standards for the ERP ecosystem.

The objective is to maintain:

* high responsiveness
* scalable performance
* optimized rendering
* efficient resource usage
* low latency
* maintainable optimization
* enterprise-grade scalability

Performance optimization must never compromise architecture quality or maintainability.

---

## Core Performance Principles

Always prioritize:

* efficient rendering
* low memory usage
* scalable architecture
* reusable logic
* optimized state management
* modular performance optimization

Avoid premature optimization that harms readability or maintainability.

---

## Rendering Optimization Rules

Always optimize:

* component rendering
* render frequency
* render tree size
* conditional rendering
* UI updates

Avoid:

* unnecessary re-renders
* oversized render trees
* duplicated rendering logic
* deeply nested rendering structures

Components must remain lightweight and modular.

---

## State Performance Rules

State management must remain optimized.

Always:

* isolate local state when possible
* centralize shared state responsibly
* minimize unnecessary state updates
* avoid duplicated state

Avoid:

* uncontrolled global state
* excessive shared state
* deeply nested state structures
* unnecessary reactive updates

---

## API Performance Rules

Always optimize:

* request frequency
* payload size
* response handling
* request reuse

Avoid:

* duplicated requests
* unnecessary polling
* repeated data fetching
* oversized payloads

Prefer centralized service layers.

---

## Lazy Loading Rules

Always use lazy loading when appropriate for:

* pages
* large modules
* heavy components
* reports
* charts
* large datasets

Avoid loading unnecessary resources during initial rendering.

---

## Memory Management Rules

Always:

* remove unused listeners
* clean subscriptions
* destroy unused references
* prevent memory leaks

Avoid:

* orphan listeners
* persistent unused states
* unnecessary caching
* uncontrolled intervals

---

## Event Optimization Rules

Always:

* centralize event handling
* debounce expensive actions
* throttle repetitive events when necessary

Avoid:

* duplicated listeners
* uncontrolled event propagation
* excessive reactive triggers

---

## Component Performance Rules

Components must remain:

* small
* isolated
* reusable
* lightweight

Avoid:

* giant components
* deeply nested components
* overloaded render logic
* duplicated UI processing

Always split oversized components.

---

## List Rendering Rules

Optimize rendering for:

* tables
* lists
* grids
* large datasets

Always:

* paginate large datasets
* virtualize heavy lists when necessary
* avoid rendering hidden content unnecessarily

---

## Database Performance Rules

Always optimize:

* queries
* indexing
* filtering
* sorting
* relational access

Avoid:

* unnecessary joins
* repeated queries
* unindexed searches
* oversized database operations

---

## Service Optimization Rules

Services must remain:

* reusable
* isolated
* optimized
* centralized

Avoid:

* duplicated transformations
* repeated business operations
* excessive processing inside UI

---

## Asset Optimization Rules

Optimize:

* images
* icons
* fonts
* bundle sizes
* static assets

Avoid:

* oversized assets
* duplicated assets
* unnecessary dependencies

---

## Import Optimization Rules

Always:

* remove unused imports
* avoid importing entire libraries unnecessarily
* organize dependencies efficiently

Avoid dependency bloat.

---

## Logging Rules

Logging must remain controlled.

Avoid:

* excessive console logging
* unnecessary production logs
* performance-heavy debugging

Production environments must remain clean and optimized.

---

## Scalability Performance Rules

All performance decisions must consider:

* future scaling
* future modules
* larger datasets
* more users
* enterprise growth

Never create optimization strategies that block scalability.

---

## Performance Monitoring Rules

The ERP must maintain visibility over:

* rendering bottlenecks
* memory usage
* API performance
* state complexity
* database performance

Performance problems must remain traceable.

---

## Performance Documentation Rules

Always maintain updated:

* PERFORMANCE_GUIDELINES.md
* DATA_FLOW_MAP.md
* MODULE_NETWORK_MAP.md

Performance architecture must remain documented.

---

## Architecture Preservation Rules

Performance optimization must never:

* break architecture
* reduce maintainability
* create hidden dependencies
* bypass modular structure

Maintain architecture integrity during optimization.

---

## Final Objective

Transform the ERP into:

* a high-performance ecosystem
* a scalable enterprise platform
* an optimized AI-friendly architecture
* a responsive operational system
* a maintainable long-term solution

The ERP must remain fast, scalable and efficient during continuous growth.
